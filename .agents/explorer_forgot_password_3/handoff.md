## 2. Logic Chain
1. **Observation (Services/Controller)** shows that `forgotPassword` generates a `resetToken` and saves it in the database while returning it to the client. The client then passes this token to `resetPassword` to authenticate. The generated OTP is not validated in the backend reset process.
2. **Security Leak**: Because the `resetToken` is generated by the server and immediately returned to the client in the API response, any client can bypass the OTP screen as long as they have the `resetToken`.
3. **Remediation**:
   - Storing the OTP directly in `user.resetPasswordToken` on `forgotPassword` and removing `resetToken` makes the OTP the primary validator.
   - Updating `resetPassword` to require `(email, otp, newPassword)` allows the backend to query the user by email (using `UserRepository.findByEmail`), retrieve the stored OTP, and compare it directly, ensuring actual validation of the OTP.
   - The database schema (type `String` for token, `Date` for expiry) natively supports storing 6-digit OTP strings and datetime objects without any migration.
   - The frontend can transition safely by passing `email` instead of `resetToken` in the React Router state and rendering an input field for the 6-digit OTP on the `ResetPasswordPage`.

---

## 3. Caveats
- **Brute-Force Vulnerability**: Since OTP is 6 digits, an attacker could try up to 1,000,000 combinations within 15 minutes. A Rate Limiter is recommended for `/api/v1/auth/reset-password` or a counter like `otpAttempts` to lock the account/invalidate OTP after 5 failed tries.
- **Reloading ResetPasswordPage**: Reloading (F5) clears `location.state`, causing the user to redirect back to `/forgot-password`. This is acceptable for security.

---

## 4. Conclusion
The proposed strategy replaces `resetToken` verification with direct OTP validation. This satisfies all security requirements, maintains interface contracts defined in `SCOPE.md`, and is compatible with the database schema.

## 5. Verification Method
- **Implementation Inspection**: Verify that the files `auth.service.js`, `auth.controller.js`, `ForgotPasswordPage.jsx`, and `ResetPasswordPage.jsx` match the changes detailed in `analysis.md`.
- **E2E verification**:
  1. Trigger `/api/v1/auth/forgot-password` with email, check the console output to obtain the generated OTP, and verify that the API response data does NOT contain a `resetToken`.
  2. Send `/api/v1/auth/reset-password` with the correct email, OTP, and new password. Verify status code 200.
  3. Attempt `/api/v1/auth/reset-password` with an incorrect OTP. Verify that it returns status code 400 with an invalid OTP message.
